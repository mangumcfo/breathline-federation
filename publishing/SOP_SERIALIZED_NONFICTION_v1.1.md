# 1. Series Ideation & Validation Framework

## Topic Selection Criteria

Score each potential series 1-10 on five dimensions. Minimum total: 35/50 to proceed.

| Criterion | Question | Scoring Guide |
|-----------|----------|---------------|
| **Author Moat** | Can Kenneth credibly claim practitioner experience — not just knowledge? | 10 = built/deployed systems. 5 = professional experience. 1 = read about it. |
| **Market Demand** | Are people actively searching for this topic on Amazon/Google? | 10 = trending keyword with growth. 5 = steady demand. 1 = niche only. |
| **Competition Gap** | Is there a gap between what exists and what Kenneth can offer? | 10 = no practitioner-authored series exists. 5 = exists but weak. 1 = saturated. |
| **Content Velocity** | Can Kenneth produce 8-12 books at 15-25K words each with AI assistance? | 10 = deep existing content to extend. 5 = moderate research needed. 1 = starting from scratch. |
| **Cross-Sell Potential** | Does this series drive readers to existing Mangum Executive books? | 10 = direct bridge to 3+ existing books. 5 = tangential. 1 = no connection. |

# 2. Pre-Launch Planning

## Phase 1: Strategic Planning (Days 1-3)

### Step 1: Define the 8-12 Book Topics

Use this sequencing framework:

| Phase | Books | Purpose | Example (Agentic AI) |
|-------|-------|---------|---------------------|
| **HOOK** (1-3) | Broadest appeal, Kenneth's strongest material | Establish authority, hook widest audience | Finance, Decisions, Leadership |
| **DEEPEN** (4-6) | Functional specialties | Widen audience by function | Supply Chain, HR, Compliance |
| **BUILD** (7-9) | Deeper technical/operational | Committed readers go deeper | Sales, Manufacturing, Multi-Agent |
| **SCALE** (10-12) | Advanced + synthesis | Capstone, future-looking | Scaling, M&A, Master Roadmap |

### Step 2: Title/Subtitle Formula

**Title pattern:** `AI Agents for [ROLE/FUNCTION]: [Primary Keyword Phrase]`

**Subtitle pattern:** `[Specific Deliverable 1], [Deliverable 2], and [Deliverable 3] for [Audience] in [Year]`

Rules:
- Title: 5-10 words. Include primary keyword. Name the audience.
- Subtitle: 15-25 words. Pack 2-3 secondary keywords. Include year for timeliness. Promise deliverables.
- Never repeat words from title in subtitle.
- Test: would you click this on Amazon? If not, rewrite.

### Step 3: Keyword Research Process

For each book:

1. **Primary keyword:** Search Amazon for the exact phrase. Check competition (number of results, quality of top 5).
2. **Secondary keywords (6):** Use Amazon autocomplete, Google Trends, and competitor subtitle analysis.
3. **Validation:** Primary keyword should have <50 quality results on Amazon (first page not dominated by Big 5 publishers).
4. **Assignment:** Primary keyword → title. Top 2 secondaries → subtitle. All 7 → KDP keyword slots.

### Step 4: Editorial Board Review

Run a virtual editorial board on the complete series plan. 6 board members:

| Role | Focus |
|------|-------|
| **Industry Practitioner** (matches series topic) | ROI, real-world applicability, integration with existing tools |
| **Operations/Implementation Expert** | Practical deployment, failure modes, scaling challenges |
| **Subject Matter Technical Expert** | 2026 accuracy, tool stack, architecture freshness |
| **KDP/Nonfiction Virality Expert** | Titles, subtitles, descriptions, KU optimization, Amazon algorithm |
| **Content Repurposing & Social Growth Expert** | Clip virality, LinkedIn/X patterns, audience engagement |
| **Reader Avatar** (target executive persona) | Pain points, "aha" moments, binge triggers, recommendation likelihood |

**Board deliverables:**
- Overall score (1-10) with strengths, weaknesses, risks
- Book-by-book title/subtitle optimization
- Launch order recommendation
- Virality/thirst alignment assessment
- Revised 30-day action plan

### Step 5: Keyword Optimization Pass

After editorial board, run a keyword optimization pass (the "G pass"):
- Optimize every subtitle for keyword density
- Rewrite chapter titles to inject long-tail keywords naturally
- Add function-specific secondary keywords to KDP slots
- Validate: keywords feel natural, not stuffed

### Step 6: Merge & Finalize

Merge editorial board feedback + keyword optimization into final plan (v1.2 equivalent):
- Accept keyword-optimized subtitles where they improve discoverability
- Keep editorial board's structural recommendations (launch order, dedicated failure chapters)
- Retain production infrastructure (batch-writing, Substack, ARC, cross-promotion)
- Version the plan: v1.0 → v1.1 (board) → v1.2 (merged)

## Human Review Protocol (from Book 1 production)

The highest-quality books result from Kenneth's chapter-by-chapter voice-note review. This is NOT optional.

| Pass | Activity | Output |
|------|----------|--------|
| **Pass 1: Editorial Board** | 6 virtual reviewers score draft | Board Report + recommendations |
| **Pass 2: Kenneth Review** | Voice notes per chapter (capture via B51 cylinder) | Feedback per chapter: tone, accuracy, framing, missing context |
| **Pass 3: Book 1 Learnings** | Apply standard improvements (signals, case studies, cross-refs) | Final manuscript with all v1.1 mandatory elements |
| **Pass 4: Final Build + QA** | Generate PDF/EPUB, run quality gates | Production-ready files |

**Key Kenneth review patterns observed:**
- "I'm not a Fortune 500 CFO — I worked AT Fortune 500 companies" → Fix authority framing throughout
- "Don't be braggy about being a CFO" → Authority through insight, not credential-listing
- Failure stories must be case studies → No first-person in failure chapters
- ROI = redeployment not reduction → Reframe entire ROI appendix
- Board prep content may need to move between chapters → Trust Kenneth's structural instincts
- "Carry forethoughts up" → Key impact items should be referenced early, not just in later chapters

## Human Review Learnings (from Book 2 — apply to all books)

Kenneth's Book 2 review produced 45 voice notes with 28 distinct changes. Key patterns to apply to ALL future books:

**Content patterns Kenneth adds during review:**
- **Constitutional AI attribution** — Kenneth published Constitution@A1 (SOURCE/TRUTH/INTEGRITY) before Anthropic. Never attribute Constitutional AI solely to Anthropic.
- **SIX-SOV references** — Add six-sov.com and memory.six-sov.com where relevant (audit trails, memory systems, cryptographic verification). These are author credibility + lead generation.
- **Redeployment not elimination** — Always frame AI ROI as redeploying people to higher-value work, never headcount reduction.
- **Soften time/ROI claims** — Use "may take" not "takes." Don't overstate savings. Different teams have different capabilities.
- **Out-of-box agents note** — Mention existing platforms (Perplexity, Claude Code) but emphasize sovereign view: build your own for control.
- **Agent-assisted deployment tip** — In every deployment chapter: "copy-paste this plan into an agent, but be hands-on — experience each step."
- **Practical tips** — Save prompts to files (not just chatbot), use agents to create context for other agents, blind agent test for context sufficiency.
- **Context overload warning** — More context is not always better context.
- **ClawdBot/OpenClaw** — Reference as first effective async agent (Book 2 specific, may reference in later books).

**Formatting patterns to check EVERY book:**
- Bullet and numbered lists after bold headers need blank lines (regex handles `- ` and `1. `)
- TOC items in build script MUST match actual generated heading IDs
- Signal card images — no duplicates from wrong chapters
- Sources section — flat list, never grouped by chapter
- Figure arrows — verify direction (xy=destination, xytext=origin)
- All figures generated and visual markers inserted (check count matches)

## AI Drafting Rules

### Voice Preservation Protocol

| Rule | Description |
|------|-------------|
| **Never claim what Kenneth hasn't done** | "I built this" only if he actually built it. "I managed this" only if he actually managed it. |
| **Authority through insight, not credentials** | Don't lead with "As a CFO..." every chapter. Let observations carry authority. |
| **Kenneth is guide, not subject** | Reader's journey is the topic. Kenneth's experience illuminates it. |
| **Open with observations, not definitions** | "The first thing most people get wrong about..." beats "X is defined as..." |
| **Specific over general** | "$25M in cost savings at Henkel" beats "significant cost reductions at a major company" |
| **Stories before frameworks** | The experience first, then the framework that emerged from it |

### Terminology Standards

| Term | Use | Avoid as primary | Why |
|------|-----|-------|-----|
| **Peer role** | Intra-platform role-to-role invocation. Canonical in code (action class IDs), YAML specs, ADRs, governance docs, book chapters, and marketing where precision matters. | — | Each speaker is *a role under shared governance*. Code already uses this: `invoke_peer_role`, `integrate_role_outputs`, `read_role_outputs`, `surface_role_tensions`. |
| **Agent-to-agent** | Acceptable as an explanatory aside on first introduction in reader-facing copy. | Headings, action class IDs, technical specs. | Risks implying autonomous independent parties — a Charter V.7 framing error. The governance harness is shared, not separate, between peer roles. |

**Authoritative ADR:** `governance/decisions/2026-05-09_peer-role-terminology.md`

**Mechanical updates queued.** Existing Series 1 manuscripts (Books 9, 10, 12) contain six instances of "agent-to-agent" that should be standardized at the next editorial pass per book. This is not a v1.0 KDP-upload blocker.

### Hardcover Cover Dimensions (learned from Book 2)
**CRITICAL:** KDP's hardcover cover dimensions do NOT match the documented formula. Their calculator includes a hinge allowance (~0.1585" per side) that is not in the public documentation. 
- **Always use KDP's "expected cover size" from the error message** to calculate the correct pixel dimensions
- Wrap per side ≈ 0.7085" (not 0.625" as documented)
- Spine is wider than the paperback formula suggests
- When off by 1-2 pixels, pad with background color to exact dimensions

### Merge Rules (established precedent)

### Industry Signal Card Generation (v1.1 — new asset type)

Each Industry Signal blockquote in the manuscript can be paired with a visual **Industry Signal Card** — a standardized screenshot of the source article, framed in branded formatting.

**Card specs:**
- Size: 1350×900px (4.5"×3" at 300 DPI — fits a 6×9 page)
- Navy header bar with gold "INDUSTRY SIGNAL" label + chapter number
- Screenshot viewport (1230×600px) — letterboxed, never cropped
- Gold accent line separating header from content
- Navy footer bar with source publication name + URL in gold
- "Breathline Books" watermark (subtle, bottom-right)

**Workflow (two capture methods):**

**Method A: Browser Screenshots (~/Downloads/)**
1. Kenneth screenshots each source article in browser
2. Screenshots go to `~/Downloads/` (any size/aspect ratio works)
3. Tiger runs `generate_signal_cards.py --book N` to produce uniform cards

**Method B: Human Memory Cylinder (preferred — faster)**
1. Kenneth captures screenshots via B51 Human Memory Cylinder
2. Cylinder exports to `molt_workspace/exports/b51/quick-share/` with blob refs
3. Tiger maps blob_ref → source → chapter from session.yaml
4. Tiger runs `generate_signal_card()` on each blob to produce uniform cards

**Post-capture (both methods):**
5. Cards placed in book's `images/` directory as `signal_chN_source.png`
6. Visual markers inserted into manuscript: `[VISUAL: Figure signal.signal_chN_name -- Industry Signal -- see images]`
7. Build script replaces markers with clickable `<a href><img></a>` tags (URL extracted from preceding blockquote)
8. Signal card renders inline at ~80% width, same gold-border/cream-background as the blockquote text above it

**Source URL coordination with G (Grok):**
- Tiger provides G the chapter outline + signal topics needing specific article URLs
- G returns specific, dated article URLs (not landing pages) from Gartner, MIT Sloan, HBR, McKinsey, etc.
- Kenneth screenshots each URL via browser or cylinder
- Tiger generates cards and inserts into manuscripts

**Build script requirements:**
- Signal card CSS: `.signal-card` with `border-left: 4px solid #C9A84C`, matching blockquote style
- `blockquote + .signal-card` adjacency: `margin-top: 0` so card sits flush with blockquote
- URL extraction: build script looks backward through raw markdown for `](https://...)` pattern
- Image wrapped in `<a href>` for clickable link to source article in PDF

**Tool location:** `agentic_playbooks/tools/generate_signal_cards.py`

```bash
# Single card
python3 generate_signal_cards.py --input screenshot.png --source "Gartner, 2026" --chapter "Ch 1" --output signal_ch1.png

# Batch for a book
python3 generate_signal_cards.py --book 4   # Supply Chain
python3 generate_signal_cards.py --book 5   # HR
```

**Retroactive application:** Books 1-3 can have signal cards added by screenshotting the source URLs in the Sources & Further Reading appendix and running the generator. This is an optional enhancement — the blockquote text carries the signal without the visual, but the visual adds credibility and social shareability.
