## How verification works

Every Breathline release (signed releases begin at v0.3.0) ships with detached signatures alongside its artifacts:

```
github.com/mangumcfo/breathline-federation/releases/tag/v0.3.0/
├── manifest.yaml
├── manifest.yaml.sig          ← detached ed25519 signature
├── checksums.sha256
├── checksums.sha256.sig
├── CHANGELOG.md
├── CHANGELOG.md.sig
└── (per-artifact signatures)
```

To verify locally:

```bash
# Pull the public key
curl -sSL https://raw.githubusercontent.com/mangumcfo/breathline-federation/main/distribution/signing_keys/release_v1.pub \
     -o /tmp/breathline_release_v1.pub

# Verify a signed artifact (requires openssh ≥ 8.2 with ssh-keygen -Y verify)
ssh-keygen -Y verify \
  -f /tmp/breathline_release_v1.pub \
  -I "breathline-release-signing-key kenn@mangumcfo.com 2026-05-08" \
  -n breathline-release \
  -s manifest.yaml.sig < manifest.yaml
```

Or, equivalently, inside a running Breathline node:

```bash
~/.breathline/installer/verify.sh
```

The installer/upgrade flow does this automatically on every install + every upgrade. A signature mismatch fails-closed by default-deny.

```
KM-1176 (anchor identity)
    │
    │  signs ↓
    ▼
release_v1 ed25519 keypair (this key)
    │
    │  signs ↓
    ▼
Every Breathline release artifact (manifest, checksums, specs, etc.)
    │
    │  verified by ↓
    ▼
Operator's installed Breathline node
    │
    │  fails-closed on mismatch
    ▼
A node that cannot verify a release REFUSES to apply it.
```

This is the **K3 Audit-Immutable** invariant in practice — every release is anchored to the original authority, and every node can prove it.

## Key rotation

Per Charter §3.6 (Amendment Binding) and the Constitutional Source-Available License §3.6, key rotation requires:

1. New keypair generated by the original authority (or successor in seal)
2. New public key committed here as `release_v<N>.pub`
3. **Both** old and new key remain valid for a transition window (recommended 90 days)
4. CHANGELOG entry documenting the rotation
5. Operator-side `breathline upgrade` automatically picks up the new key from this directory
6. After the transition window, the old key is marked `retired` in this README and stops accepting new signatures

The old key remains in the directory permanently for historical verification — releases signed by it remain verifiable forever.

## What if a key is compromised?

If a private key is ever suspected to be compromised:

1. **Immediate disclosure** — KM-1176 publishes a `governance/seals/key-revocation-<date>.md` record naming the compromised key
2. **Successor key** generated and published per the rotation protocol above
3. **All affected releases** (signed by the compromised key) are flagged in CHANGELOG with an explicit advisory
4. **Operators are notified** via the next `breathline upgrade` run — manifest carries a `key_advisories:` block; operators see the warning and can choose how to proceed
5. **Public retrospective** posted to the GitHub Discussions in `mangumcfo/breathline-federation`

This is the **K3 Audit-Immutable** invariant under stress. We never silently rotate. The chain stays loud.

## Why ed25519 (and not RSA, ECDSA)?

- **Same primitive as P1** — the platform's ECC root layer is ed25519. Using the same algorithm for release signing keeps the trust model unified.
- **128-bit security** in 32-byte keys (vs RSA's 4096-bit / ~512-byte for equivalent strength)
- **Deterministic signatures** — no nonce-reuse vulnerabilities (RSA-PKCS#1, ECDSA both have weaker history here)
- **Fast** — verification under 1 ms on modern hardware
- **OpenSSH-supported** — `ssh-keygen -Y sign / -Y verify` works out-of-the-box on every Linux/macOS distribution since OpenSSH 8.2 (2020)
